## PART A — Data Journalism Pipeline

### Tesis

Tenemos datos que ningún medio de comunicación en España puede generar por sí mismo: 180M de registros de precios de carburante (CNMC, diarios desde 2016), 500K accidentes geocodificados con detalle de vía/km/condiciones (DGT 2019-2023), 7 años de registros climáticos diarios de 900 estaciones AEMET. Cada artículo que publicamos es **primicia metodológica**: citamos la query SQL (anonimizada) y la fuente oficial. Eso es lo que atrae enlaces de elpais.com, abc.es y Wikipedia — no el SEO, sino la verificabilidad.

El moat es asimétrico: un periodista de El País tardaría semanas en hacer el análisis que nosotros generamos en segundos. La estrategia no es competir con los medios — es ser su fuente de datos.

| T6 | Las 10 horas más baratas para repostar en España | `CNMCFuelPrice` (patrón intra-semana) | Semanal | Bajo |
| T7 | Ferry más puntual del Mediterráneo: análisis 2025 | `PortCall` × `Voyage` × `FerryRoute` | Anual | Bajo (nicho alto impacto) |

### Pipeline de Generación

```
Lunes 04:00 CET — cron en contenedor `daily`
  │
  ├─ 1. Dispatch: TASK=data-journalism-generator
  │      Lee tabla `ArticleSchedule` (nuevo modelo, ver abajo)
  │      Para cada template con next_run <= now:
  │
  ├─ 2. Query Layer
  │      Ejecuta vista SQL pre-agregada (NO tabla raw)
  │      Vistas: `v_cnmc_cheapest_by_province`, `v_accident_hotspots`,
  │              `v_climate_extremes`, `v_renfe_delay_rankings`
  │      Freshness: 1h (Redis cache key por vista + parámetros)
  │
  ├─ 3. Prose Layer
  │      Claude Haiku 4.5 — prompt estricto (ver sección siguiente)
  │      Temp 0.3, max_tokens 900, timeout 15s
  │      Input: JSON estructurado con los datos + metadata del template
  │      Output: { title, summary, body, faq[] }
  │
  ├─ 4. Upsert en Prisma `Article`
  │      isAutoGenerated: true
  │      status: DRAFT (si es primer artículo del template) | PUBLISHED (recurrente ya aprobado)
  │      editorialWeight: según template (T2 evergreen = 10, T9 weekly = 7)
  │      slug: generado deterministamente (ej: gasolineras-baratas-madrid-2026-04)
  │
  └─ 5. Notificación
         Si status=DRAFT → POST a webhook Slack #editorial-queue
         Si status=PUBLISHED → ping Google Indexing API + news-sitemap revalidate
```

**Event-driven path** (paralelo al cron): cuando un colector detecta un umbral anómalo (precio gasolinera bate récord histórico, temperatura X ciudad rompe máximo de la serie), emite un evento interno → se activa el template correspondiente inmediatamente, sin esperar al lunes.

### Prompt Template (LLM Grounding Estricto)

```
Eres un periodista de datos del medio trafico.live. Escribes en español peninsular formal pero accesible. Tu única fuente es el JSON de datos que se te proporciona a continuación. NO puedes introducir información que no aparezca en ese JSON. NO puedes especular sobre causas. Si los datos no explican algo, di "los datos disponibles no permiten concluir".

DATOS:
{json_data}

INSTRUCCIONES:
1. Escribe un artículo de entre 320 y 380 palabras con este estructura exacta:
   - Párrafo de apertura (lede): la cifra más impactante del análisis en la primera frase.
   - Cuerpo (3-4 párrafos): desarrolla el ranking/hallazgo principal. Cita la fuente de datos al menos 3 veces usando la forma "según datos de {source}".
   - Sección "Metodología" (50 palabras): describe qué datos se usaron, el período temporal, y la métrica exacta. Termina con "Datos disponibles en trafico.live/api".
   - Sección "Preguntas frecuentes": 3 preguntas y respuestas breves basadas EXCLUSIVAMENTE en los datos.

2. El título debe contener el número de la cifra principal (ej: "Las 5 provincias...", "El día 14 de julio...").
3. Tono: periodístico, sin alarmismo, sin adjetivos valorativos sin respaldo en dato.
4. Formato de salida: JSON estricto con campos { "title": str, "summary": str (1 frase), "body": str (HTML mínimo: <p>, <ul>, <strong>), "faq": [{"q": str, "a": str}] }.
```

### Estrategia de Citación (E-E-A-T)

Cada artículo publicado incluye una sección "Metodología" fija con:
- Fuente oficial (CNMC, DGT, AEMET, Renfe)
- Período temporal exacto
- Métrica y unidad
- Query SQL anonimizada (sin credenciales, solo la lógica SELECT)
- Enlace al endpoint API público correspondiente (`/api/combustible/historico`, `/api/accidentes/microdata`, etc.)

Esto convierte cada artículo en **recurso verificable**. Un redactor de elpais.com puede copiar la query y validar el dato. Wikipedia acepta fuentes verificables con metodología explícita. Los organismos oficiales (DGT, CNMC) tienden a enlazar análisis que citan sus datos correctamente.

Schema.org por artículo:
- `NewsArticle` (publisher: Organization trafico.live, author: Organization + human reviewer name si pasó revisión editorial)
- `Dataset` en artículos de ranking (describe el dataset subyacente con `variableMeasured`, `temporalCoverage`)
- `FAQPage` (wired sobre el array `faq[]` generado por Haiku)
- No usar `HowTo` en artículos de datos — es semánticamente incorrecto y Google lo penaliza si el contenido no tiene pasos de instrucción.

### Coste y Escala

- 50 artículos/día × ~1.800 tokens input+output × $0.80/MTok (Haiku 4.5) = **$0.072/día**
- 1.500 artículos/mes = **$2.16/mes**
- Con revisión humana del top 20% viral (300 artículos/mes): 2h/semana de editor

Escalado sin coste adicional: las vistas SQL pre-agregadas ya absorben la carga; Haiku en batch mode con `anthropic.messages.batches` reduce coste otro 50%.

### Outreach a Medios

Kit pre-construido activado automáticamente cuando un artículo alcanza el umbral de virabilidad (template T2, T3, T5 — impacto alto):

1. El collector detecta "artículo con potencial de cobertura" (ranking nacional, récord histórico)
2. Genera borrador de email usando `services/collector/tasks/social-broadcast/` como base
3. Email a lista de contactos de medios: asunto `[DATOS EXCLUSIVOS] {título artículo} — trafico.live`
4. Cuerpo: 3 párrafos — cifra principal, metodología resumida, enlace al artículo con `utm_source=press`
5. Adjunto: CSV con los datos raw del ranking (aumenta probabilidad de uso editorial)

Destinatarios objetivo: sección de datos/economía de elpais.com, elmundo.es, 20minutos.es, abc.es, lavanguardia.com, cincodias.elpais.com. Contactos se gestionan en lista interna (no automatizar cold outreach masivo — máximo 5 medios por artículo, selección manual inicial).

## PART B — Events + Mobility Cross

### Tesis

Nadie en España cruza calendario de eventos con datos de movilidad en tiempo real. Google Maps da ETA sin contexto del evento. Waze idem. Moovit da transit sin referencia al partido. El resultado: 80.000 personas van al Bernabéu el mismo domingo y ninguna aplicación les dice "sal a las 18:30, no a las 20:00, porque a las 20:00 el metro cierra y la A-6 colapsa".

La oportunidad es doble: SEO masivo pre-evento (búsquedas "cómo llegar Bernabéu partido hoy" pican 24-48h antes) + utilidad real-time que genera retorno directo y afiliado.

---

### Taxonomía de Eventos

| Categoría | Ejemplos | Volumen consultas | Recurrencia | Dificultad datos |
|---|---|---|---|---|
| Fútbol LaLiga / Champions | Real Madrid-Barça, derbi, europeos | 50K-500K/evento | Semanal | Baja — API LaLiga free tier |
| Conciertos masivos | Mad Cool, Primavera Sound, tour artistas | 10K-100K/evento | Irregular | Media — ver nota legal |
| Operaciones DGT | Puente mayo, Semana Santa, verano, Navidad | 200K-1M/operación | Anual predecible | Muy baja — datos públicos |
| Ferias y congresos | MWC Barcelona, Fitur, Fruit Attraction | 5K-50K/evento | Anual | Baja — calendarios públicos |
| Cultural / festivo | Fallas, Sanfermines, Semana Santa procesiones, Feria Abril | 20K-200K/evento | Anual | Baja — fechas fijas |
| Huelgas transporte | AENA, Renfe, CRTM, tripulantes | 50K-300K/evento | Irregular | Media — fuentes oficiales |
| Manifestaciones | "manifestación madrid hoy" (27K/mes, KD~0) | 27K/mes constante | Irregular | Alta — ver nota legal |

**Nota legal — conciertos y manifestaciones:**

Scraping de Ticketmaster/LiveNation viola sus ToS y expone a demandas DMCA. La vía correcta es afiliado directo: SeeTickets ES tiene programa de afiliados documentado; Entradas.com también. Con afiliado obtienes feed de eventos estructurado + comisión por venta (€1-3/ticket). Integrar primero SeeTickets, que tiene mayor penetración en festivales españoles. Ticketmaster solo para eventos de alta demanda con URL directa (no scraping — deeplink afiliado).

Para manifestaciones: la Delegación del Gobierno publica comunicaciones previas en su web. No son estructuradas pero son scrapeable legalmente (contenido administrativo público). En Sprint 11, implementar como lista curada manual para las 10 ciudades principales, con scraping opcional como mejora posterior. El intent "manifestación madrid hoy" (KD~0, 27K/mes) es ganancia pura con una página de actualización diaria.

### Cálculos de Movilidad por Evento

Para cada evento, la página calcula en el servidor (SSR con revalidate bajo):

1. **ETA al recinto** — para los 10 barrios/municipios con mayor volumen de asistentes histórico (estimado por `TrafficIntensity` picos post-evento)
2. **Carga tráfico actual** — cross con `TrafficIncident` + `TrafficIntensity` en radio 5km del recinto
3. **Alternativas transit** — cross con `TransitStop` + `TransitRoute` en radio 800m (ya en Prisma)
4. **Ventana de salida óptima** — "Si sales antes de las X llegas en Y min, después de las X+2h llegas en Z min" (derivado de `HourlyTrafficProfile` del día de semana correspondiente)
5. **Predicción volver** — hora de fin del evento + ventana post-evento por tipo (fútbol: +45min pico, concierto: +20min, procesión: gradual)
6. **Parking** — deeplink Parclick con URL del recinto como parámetro (comisión €8-15/reserva)

Estos cálculos NO requieren un modelo ML nuevo en Sprint 11 — son queries sobre datos que ya existen en Prisma, con lógica determinista.

### Estructura de URLs

```
/evento/[slug]                          # Per-event, generado 48h antes
  └─ real-madrid-atletico-2026-05-04

/estadio/[slug]/como-llegar             # Evergreen por recinto (nunca expira)
  ├─ bernabeu/como-llegar
  ├─ metropolitano/como-llegar
  └─ montjuic/como-llegar

/operacion-especial/[slug]              # DGT ops anuales
  ├─ puente-mayo-2026
  ├─ semana-santa-2026
  └─ operacion-verano-2026

/huelga/[operator]/[date]               # Huelgas confirmadas
  └─ renfe/2026-05-15

/manifestacion/[ciudad]/[date]          # Manifestaciones comunicadas
  └─ madrid/2026-05-01
```

El slug del evento es deterministico y predecible: `{home-team}-{away-team}-{YYYY-MM-DD}`. Esto permite que la URL exista antes de que el evento ocurra (indexación previa) y que sea reutilizable si el partido se repite en siguiente temporada con redirect 301 al nuevo slug.

### Flujo de Contenido por Evento

```
T-48h  → Artículo "Guía completa para llegar a [Partido/Concierto]"
           - ETA por barrio, parking, transit, restricciones DGT si aplica
           - Auto-generado por Claude Haiku con datos del recinto + evento

T-2h   → Actualización en página /evento/[slug]
           - "Tráfico ahora: 35 min al estadio por A-6 (habitualmente 20 min)"
           - Incidencias activas en radio 5km (TrafficIncident live)
           - Recomendación: "Sale ahora o en 90 min"
           - SSR con revalidate: 120s durante las 3h pre-evento

T+1h   → Artículo de regreso "Cómo volver de [evento]: mejores rutas y horarios metro"
           - Generado automáticamente post-evento
           - Datos reales del evento (incidencias reales, aforo según fuente)

T+24h  → Artículo de datos "Así afectó [partido/festival] al tráfico de Madrid"
           - Cross `TrafficIntensity` antes/durante/después del evento
           - Fuente: datos propios — artículo 100% exclusivo
```

### Moat Competitivo

- **Google/Waze**: dan ETA sin contexto del evento. No saben que hay 80.000 personas yendo al mismo sitio.
- **Moovit**: transit sin cruce de tráfico. No predice saturación.
- **LaLiga app**: información del partido, sin movilidad.
- **Timepot.com** (PL, F1-focused): no opera en España, no cubre LaLiga ni cultura.

El cruce evento×movilidad×tiempo real×afiliados no existe en España. Es un producto, no solo contenido — y eso lo hace difícil de replicar aunque alguien lo intente.

```prisma
model TrafficEvent {
  id          String      @id @default(cuid())
  slug        String      @unique
  title       String

  status      EventStatus @default(SCHEDULED)

  @@index([category, startAt])

  id        String   @id @default(cuid())
  slug      String   @unique
  name      String
  city      String

}

  SCHEDULED LIVE FINISHED CANCELLED
}
```

Collector nuevo: `TASK=event-sync` — cron diario, ingesta desde LaLiga API + SeeTickets feed + DGT operaciones (scraping legal de página pública). Para Sprint 11 puede ser lista curada de 50 eventos hardcodeados como seed; el collector dinámico puede llegar en Sprint 12.

Rutas nuevas:
- `/evento/[slug]/page.tsx` — SSR con revalidate 120s durante evento activo, 3600s otherwise

## Relación entre Módulos

Los dos módulos se refuerzan mutuamente: un artículo de data journalism sobre "los puentes más peligrosos de España" linkea naturalmente a `/operacion-especial/semana-santa-2026`. Una página de evento linkea al artículo "Cómo afectó el Mad Cool 2025 al tráfico de Madrid". La arquitectura de `Article` ya permite `category: ANALYSIS` con province y tags — es la misma tabla para ambos flujos.

El backlink loop completo: datos exclusivos → medios los citan → autoridad de dominio sube → páginas de evento ranquean más rápido → más tráfico → más datos propios → artículos más precisos.
