**Uso:**

Útil para: redirecciones search-to-landing, sugerencias en el `<SearchOverlay>`, breadcrumb inteligente, analytics del intent.

**Cómo lo van a usar en Sprint 2+:**

Las páginas que Team C (Trenes) y Team D (Carreteras) construyan pueden añadir los slots al pie como placeholder:

```tsx
import { PredictionSlot, WeatherImpactSlot, DecisionSlot } from "@/components/layers";
import { TimestampBadge } from "@/components/freshness/TimestampBadge";

export default function StationPage({ params }) {
  // ... station data fetch (capa 0 + 1) ...

  return (
    <article>
      <h1>{station.name}</h1>
      <TimestampBadge at={lastUpdate} source="Renfe GTFS-RT" />

      {/* live board — existing Sprint 2 work */}
      <DepartureBoard data={board} />

      {/* SLOTS LISTOS PARA SPRINT 8-10 — renderizan null ahora */}
      <PredictionSlot
        state={{ status: "unavailable" }}
        title="Previsión de retraso hoy"
      />
      <WeatherImpactSlot
        state={{ status: "unavailable" }}
        entityLabel={station.name}
      />
      <DecisionSlot
        state={{ status: "unavailable" }}
        question={`¿Mi tren saldrá puntual hoy desde ${station.name}?`}
      />
    </article>
  );
}
```

Cuando Sprint 9 construya el modelo predictivo y lo conecte al endpoint `/api/forecast/train-delay`, el slot pasa de `unavailable` a `ready` y **la página se enriquece sin tocar el template**.

Esto es la decisión arquitectónica crítica del plan: ensamblar hoy los sockets de las capas que lleguen en 6 meses, para no reescribir 1.500 páginas de estación cuando lleguemos a Sprint 9.
